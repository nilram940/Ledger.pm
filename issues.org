** FIXED Account identification via filename is fragile              :bug:
:PROPERTIES:
:ID: BUG-009
:END:
:LOGBOOK:
- State "FIXED" from "TODO" [2026-04-21]
:END:

Both importers derive the account name from the statement filename:
~fromStmt~ strips the path, strips everything from the first ~-~, and strips
the extension:

#+begin_src perl
$account =~ s/-.*//;   # "Checking-2026-03.csv" → "Checking"
#+end_src

This string is then used as:
1. A key into the ~%handlers~ map to find the right callback
2. A key into the ~%csv~ config map to find field layout
3. The default account name passed to ~addStmtTran~ / ~addStmtBal~

This means filenames must follow an exact convention, and the short prefix
(e.g. ~Checking~) is used as both a config key and a ledger account name,
conflating two distinct concepts.

*** What the importers already know

- *OFX*: ~<ACCTID>~ and ~<BANKID>~ are parsed into ~$data->{acctid}~ by the
  ~acctid~ handler but are never surfaced to the callback or used for routing.
  The file contains the authoritative account identity.
- *CSV*: Account may appear as a field in the row (~account~ in ~fields~), but
  the config-dispatch key and the default account both still come from the
  filename.
- *Plaid/Teller JSON*: Structured account objects with IDs are available.

*** Prior attempt

~accounttab~ was intended to map 4-digit account-number suffixes to full
ledger account names, but ~getaccount~ was never wired into ~fromStmt~ and
remains a standalone function callers must invoke manually (see BUG-001).

*** Proposed solutions

1. *Explicit account in CSV config*: add an ~account~ key to the per-source
   config hashref.  The filename prefix remains the config dispatch key but is
   no longer used as the ledger account name:
   #+begin_src perl
   Checking => { account => 'Assets:Checking:MyBank',
                 fields  => [qw(date id payee quantity)] }
   #+end_src

2. *OFX routing table*: a YAML config mapping ~BANKID:ACCTID~ → full ledger
   account name.  ~fromStmt~ passes the parsed ~acctid~ to a lookup function
   instead of using the filename.  Subsumes ~accounttab~ with a working
   implementation.

3. *Explicit ~account~ parameter to ~fromStmt~*: simplest possible fix — let
   the caller pass the account name directly and use the filename only as a
   config-dispatch key.

Solutions 1 and 3 are complementary and low-risk; solution 2 is the proper
long-term fix for OFX but requires surfacing ~acctid~ from the OFX parser to
~fromStmt~.

4. *Preprocessor with ~LEDGERNAME:~ injection* (preferred for OFX):
   A standalone script reads the ~<ACCTID>~ from the OFX file, looks it up in a
   mapping (YAML or same ~accounttab~ format), and injects a ~LEDGERNAME:~ line
   into the OFX free-form header before Ledger.pm sees the file:

   #+begin_example
   OFXHEADER:100
   DATA:OFXSGML
   LEDGERNAME:Assets:Checking:MyBank
   ...blank line...
   <OFX>...
   #+end_example

   ~Ledger::OFX::parse~ already splits off the header block (everything before
   the first ~<~) but currently discards it.  A one-line change reads
   ~LEDGERNAME:~ from that block and passes it to the callback, which uses it
   instead of the filename-derived prefix.

   This approach mirrors the existing pattern in ~getplaid3.pl~, where the
   downloader's ~override~ stanza injects ~ledger_name~ into the JSON before
   writing to disk, so ~Ledger::JSON::Plaid~ finds it in the data without any
   special-case logic in ~fromStmt~.  The preprocessor can be a standalone
   script (one per institution, or a generic one driven by a YAML mapping) and
   keeps the injection *outside* Ledger.pm entirely.

*** Implementation

Implemented option 4 (LEDGERNAME: preprocessor approach):

- ~Ledger::OFX::parsefile~ scans the free-form header block (before the first ~<~)
  for a ~LEDGERNAME:~ line using ~/^LEDGERNAME:\s*(.+?)\s*$/m~.
- The parsed account name is stored in ~$account~ (module-level, reset to ~undef~
  at the start of each ~parsefile~ call to prevent cross-file bleed).
- ~stmttrn~ sets ~$tran{account} = $account~ when present; ~ledgerbal~ sets
  ~$balance{account} = $account~ when present.
- Investment handlers (~inv~, ~invbal~, ~invpos~) also set account from
  ~$account~ when present, so LEDGERNAME: applies to INVBUY/INVSELL
  transactions and INVPOS/INVBAL balance assertions.  ~stop()~ subsequently
  appends ~:TICKER~ to inv/invpos accounts (e.g. ~Assets:Brokerage:Fidelity:AAPL~).
- ~addStmtTran~ and ~addStmtBal~ already use ~$stmttrn->{account} || $accdef~,
  so LEDGERNAME: overrides the filename-derived default with no further changes.

Covered by:
- ~t/test_bug009.pl~ (5 assertions: banking transaction and balance assertion
  use LEDGERNAME account, not filename-derived "BankStmt" prefix).
- ~t/test_bug009_inv.pl~ (4 assertions: INVBUY account = ~Assets:Brokerage:Fidelity:AAPL~,
  INVPOS balance line uses same account, neither uses filename-derived "BrokerStmt").

** TODO idtag option is not consistently honoured                       :bug:
:PROPERTIES:
:ID: BUG-002
:END:

~Ledger->new~ accepts an ~idtag~ constructor parameter (default ~'ID'~) that is
meant to control which ledger tag stores transaction IDs.  The value is stored
in ~$self->{idtag}~ but is never read back; all of the following sites
hardcode the string ~ID:~ instead:

| File | Line | Hardcoded use |
|------+------+---------------|
| ~Ledger.pm~ | 244 | ~"ID: $key"~ — posting note written in ~addStmtTran~ |
| ~Ledger/Posting.pm~ | 30 | ~/ID:\s*(\S+)/~ — ~getid()~ regex |
| ~Ledger/XML.pm~ | 140 | ~/^\s*ID:\s*(\S+)/~ — XML import ID extraction |
| ~Ledger/CSV.pm~ | 110 | ~$csv{note}!~/ID:/~ — transfer detection heuristic |

Two possible resolutions:

1. *Thread the tag through*: pass ~$idtag~ (or a compiled regex) to each site.
   ~getid~ in ~Posting.pm~ is the most impactful since it drives ~distance()~
   ID-based matching.  The ~Ledger~ object is not in scope in ~Posting~ or ~CSV~,
   so the tag would need to be passed explicitly or stored on each posting at
   construction time.

2. *Remove the option*: ~'ID'~ appears to be the only tag in use in practice.
   If no caller has ever needed a different tag, the right fix may be to delete
   the ~idtag~ constructor parameter, hardcode ~'ID'~ openly, and document the
   limitation.  This is simpler and honest about what the code actually does.

** DONE Replace strdist with token-based recall in distance()          :bug:
:PROPERTIES:
:ID: BUG-003
:PRIORITY: high
:END:

~distance()~ uses ~strdist~ to compare payees: it truncates the incoming
statement payee to the length of the existing ledger payee and counts
character mismatches.  This works when the bank string *starts with* the
ledger name, but fails for intermediaries (Square, PayPal, etc.) where the
merchant name does not appear at position 0:

#+begin_example
ledger: "Blue Bottle"          tokens: [blue, bottle]
bank:   "SQ *BLUE BOTTLE 123"  tokens: [sq, blue, bottle]

strdist: substr(bank, 0, 11) = "sq *blue bo"
         vs "blue bottle" → 4 mismatches → distance ≈ 0.36  (wrong)

token recall: [blue, bottle] ∩ [sq, blue, bottle] / 2 = 100%
              → distance contribution 0  (correct)
#+end_example

Proposed fix: replace the ~strdist~ block in ~distance()~ with a token-recall
score — the fraction of the ledger payee's tokens that appear in the statement
payee's tokens.  A recall of 1.0 contributes 0 to ~$dist~; 0.0 contributes the
same weight as the current maximum strdist penalty.  The ~_tokenize~ helper
introduced for the Naive Bayes classifier is already available.

*** TODO Pending (Plaid) transactions do not clear                    :bug:

The automation matches pending transactions by ID, but Plaid's pending ID
either changes or disappears when a transaction posts.  Two failure modes:

1. *ID changes on posting*: the pending transaction has ID ~P-xxx~ and the
   posted transaction arrives with a different ID ~C-xxx~.  The importer
   creates a new cleared transaction, leaving the old pending entry as an
   orphan.

2. *Transaction disappears*: the pending entry is removed from subsequent
   Plaid exports without a posted counterpart (e.g. a cancelled hold, as in
   BUG-013).  The orphan is never matched and remains until manually removed.

Both failure modes produce orphaned pending transactions; BUG-013 (removed
holds) is essentially the same problem scoped to case 2.  A unified fix may
address both:

- *ID-change case*: after the absence scan from BUG-013 identifies a pending
  transaction no longer in the current export, attempt a date+amount match
  against newly-posted cleared transactions before deleting it.  A match
  means the pending entry should be deleted (the cleared version is
  authoritative); no match means the transaction was removed without posting
  and should also be deleted.
- *Both cases* require the ~pending_snapshot~ opt-in from BUG-013 (Plaid
  provides a complete snapshot; OFX/CSV do not).

Possible additional approach: match pending→cleared by date + amount during
normal import, falling back to ID only when an ID match fails.  Risk: date
and amount alone may not be unique enough to avoid false matches.

*** FIXED CC-side cosmetic fix: account-specific desc cache

~$self->{desc}~ restructured from a flat ~{ raw_payee → canonical }~ map
to ~{ account => { raw_payee → canonical } }~.  Two accounts that share
the same raw payee string (e.g. ~PAYMENT THANK YOU~ on both Visa and MC)
now maintain independent mappings and resolve to different canonical
names.

- ~_read_payeetab~ detects old flat-format files and wraps them under a
  ~__global__~ fallback key; new entries are always written
  account-specifically.
- The dedup write-back (~addStmtTran~ line 210) writes to both the
  account-specific bucket and ~__global__~, so unambiguous payees
  (e.g. ~Starbucks~) are still shared across accounts as a useful
  default for first-sight lookups.
- Lookup merges ~__global__~ + account-specific with account-specific
  winning on collision.

*** FIXED Bayesian classifier now predicts Equity:Transfers: destinations

~gentable~ previously excluded all ~Equity:~ destinations from training.
~Equity:Transfers:*~ entries are meaningful signal (they encode a learned
~source_account + payee_tokens → transfer_tag~ relationship) so the
exclusion was narrowed:

#+begin_src perl
# Before:
next if $dest =~ /^Equity:/;
# After:
next if $dest =~ /^Equity:/ && $dest !~ /^Equity:Transfers:/;
#+end_src

~balance()~ in ~Transaction.pm~ gained a new branch before the existing
Asset/Liability branch:

#+begin_src perl
if ($account =~ /^Equity:Transfers:(.+)/) {
    $tag = $1;
}
#+end_src

This means that once a card account has accumulated enough ~PAYMENT THANK
YOU → Equity:Transfers:Visa~ training examples, subsequent payments are
routed correctly by the classifier alone — no handler required.  Covered
by ~t/test_bug015.pl~ (3 training transactions → classifier predicts
~Equity:Transfers:Visa~ at ≥90% confidence for a new payment).

*** TODO Checking-side routing remains open

Both fixes above address the CC side.  On the checking side (e.g.
~BOA CRDT CARD PMT~ appearing for both a Visa and an MC payment), the
classifier cannot help: both transactions share the same source account
(~Assets:Checking~) and identical payee tokens, so there is no signal
to distinguish them.  The workarounds and fix options below still apply.

When two different accounts share the same raw payee string on an
importing account's statement, the desc cache can only remember one
canonical name, so all matching transactions are routed to the same
~Equity:Transfers:~ account regardless of which account they actually
belong to.  Most commonly seen on checking statements (e.g.
~BOA CRDT CARD PMT~ appearing for payments to both a Visa and an MC),
but the underlying problem affects any statement source where the bank
uses the same payee string for transactions that belong to different
accounts.

*** Concrete failure

Accounts: ~Assets:Checking~, ~Liabilities:Visa~, ~Liabilities:MC~.

Checking statement:
#+begin_example
2026/04/01  BOA CRDT CARD PMT   -$150.00  ; Visa payment
2026/04/20  BOA CRDT CARD PMT   -$200.00  ; MC payment
#+end_example

Both rows have the same raw payee.  After the first import, ~desc.yaml~
contains:
#+begin_example
BOA CRDT CARD PMT: BoA Visa
#+end_example

On the next run the MC payment is resolved via the desc cache to
~BoA Visa~, which the handler routes to ~Equity:Transfers:BoA-Visa~.
When the MC statement is then imported (with transfer tag ~BoA-MC~), the
queue key ~BoA-MC-200.00~ has no match; two orphaned ~Equity:Transfers~
stubs remain.

The same collision appears on the credit card side: both the Visa and
MC statements show a generic payment payee (e.g. ~PAYMENT THANK YOU~),
so desc maps that string to whichever card was imported first and both
payments end up with the same canonical name.  Transfer routing is
unaffected on this side because the handler is keyed by account
(~Visa => { ... }~, ~MC => { ... }~), not by payee, so the correct
~Equity:Transfers:~ tag is assigned regardless.  The symptom here is
cosmetic — wrong payee name in the ledger — rather than a broken
transfer match.

*** Root cause

Two reinforcing mechanisms:

1. *desc write-back on dedup* (~Ledger.pm~ line 210):
   #+begin_src perl
   $self->{desc}->{$payee} = $self->{id}->{$key};
   #+end_src
   When a Visa dedup occurs, ~$self->{id}->{key}~ holds the canonical
   payee ~"BoA Visa"~ (loaded by ~ledgerCSV~ from the already-written
   ledger entry).  This reinforces the mapping
   ~"BOA CRDT CARD PMT" → "BoA Visa"~ in desc on every import that
   sees an existing Visa payment.

2. *desc is payee-string-only* — there is no date or account-name
   context, so a single raw payee string can only map to one canonical
   name regardless of which account the transaction actually belongs to.

*** Workaround

Write a code-ref handler that discriminates by day of month.  Transaction
dates for distinct accounts are typically weeks apart and very stable
month-to-month, making this far more reliable than amount-based
discrimination (amounts fluctuate every statement):

#+begin_src perl
Checking => {
    'BOA CRDT CARD PMT' => sub {
        my $txn = shift;
        my $day = (localtime($txn->{date}))[3];
        return $ledger->transfer($txn,
            $day <= 10 ? 'BoA-Visa' : 'BoA-MC');
    },
}
#+end_src

A simpler alternative that avoids per-account date thresholds: use a
shared generic transfer tag on all sides (e.g. all use
~transfer => 'BoA-CC'~).  The transfer matcher already validates by
amount+date, so two transactions for different accounts whose dates
differ by more than five days cannot falsely pair even if the amounts
happen to match.  The collision window (same amount, within 5 days,
for two different accounts) is effectively impossible given typical
billing cycles.

*** Fix options

1. *Don't write desc for transfer-tagged payees*: when a handler
   provides a ~transfer~ key, the resolved payee came from the handler
   configuration, not from a true disambiguation of the raw bank string.
   Suppress the desc write-back for these transactions.  Prevents
   propagation but does not solve the ambiguous case without a handler.

2. *Date-keyed desc entries*: allow ~desc.yaml~ values to be a hash
   keyed by day-of-month range rather than a bare string, so the user
   can explicitly configure:
   #+begin_example
   BOA CRDT CARD PMT:
     "<= 10": BoA Visa
     "> 10": BoA MC
   #+end_example
   Day-of-month thresholds are more stable than amount thresholds, but
   still require per-user configuration and breaks if a payment date
   drifts across the boundary.

3. *Cross-tag transfer matching*: allow the transfer queue to match
   entries across different tags when amount+date agree and one tag is a
   prefix of another (e.g. ~BoA-CC~ matches ~BoA-Visa~).  Riskier —
   could produce false matches in other scenarios.

4. *Document the shared-tag pattern as the recommended approach*
   (lowest risk): the checking-side handler uses a single generic tag
   (~BoA-CC~); both credit card import scripts also use ~BoA-CC~.
   Transfer matching works by amount+date within the shared pool.  Two
   payments to different cards whose due dates are more than five days
   apart cannot falsely pair, making same-amount collisions practically
   impossible given typical billing cycles.

** TODO makeid collisions and account-name coupling                    :bug:
:PROPERTIES:
:ID: BUG-010
:END:

Generated ID format: ~{initials}[-!]-[salt-]{FITID | date+amount}~

~makeid()~ falls back to ~date+amount~ when no FITID is available.  This can
lead to collisions between distinct transactions that share the same date and
amount.

*** Salt mechanism (partial mitigation)

~makeid~ accepts a ~salt~ field on the transaction hash and inserts it between
the initials prefix and the FITID/fallback:

#+begin_src perl
$id.=$trdat->{salt}.'-' if $trdat->{salt};
#+end_src

Salt is not set by any parser module.  It is set in per-account CSV handler
callbacks for sources whose description field embeds a unique transaction
identifier after asterisks:

#+begin_src perl
if ($csv->{desc} =~ /\*+([^*]+)/) {
    $csv->{salt} = $1;
    $csv->{salt} =~ s/\s+/-/g;
}
#+end_src

For the CSV source this was written for, the extracted string is a stable
per-transaction identifier embedded in the description — effectively a FITID
substitute.  Setting it as ~salt~ rather than ~id~ means ~makeid~ still uses the
date+amount path but prefixes it with the unique token, producing a
collision-free key in practice.  The cleaner fix would be to set ~$csv->{id}~
directly, which would skip the date+amount fallback entirely.

CSV accounts without such a handler remain vulnerable to collision.

*** Account-name coupling

The initials prefix is derived from the last ~:~-separated component of the
account path (e.g. ~Assets:Checking:MyBank~ → ~M~).  Two problems:

1. *Rename breakage*: renaming an account changes the initials, making all
   previously written ~ID:~ tags in the ledger stale — the dedup cache no
   longer matches them on the next import.
2. *Prefix collision*: accounts whose last component shares initials (e.g.
   ~MyBank~ and ~MegaBank~ both give ~M~, or ~Credit Card~ under both ~Assets~
   and ~Liabilities~ both give ~CC~) can collide in the dedup cache across
   accounts.

*** Proposed change

Replace the account-initials prefix with a fixed source-type prefix that
identifies the import channel, not the account.  The stable per-transaction
key is unchanged where a bank-supplied FITID exists; for CSV sources without
a FITID, an MD5 digest of stable raw fields replaces the date+amount string.
See ~../scripts/ledgertemp4.pl~ for the working ~TEMP-{md5}~ prototype; compare
with ~../scripts/ledgertemp2.pl~ which uses the old initials+date+amount form.

**** New ID format

| Source  | Prefix   | Key                                                          |
|---------+----------+--------------------------------------------------------------|
| OFX     | ~OFX~    | Bank-supplied FITID (~$trdat->{id}~) — already stable       |
| CSV (no FITID) | ~CSV~ | ~md5_base64("$date\|$payee\|$quantity")~ — raw pre-callback fields |
| Plaid   | ~PLAID~  | Plaid transaction_id (~$trdat->{id}~)                        |
| Teller  | ~TELLER~ | Teller id (~$trdat->{id}~)                                   |
| Template | ~TEMP~  | ~md5_base64("$date\|$payee\|$amount")~ — stable fields only (see note) |

Result: ~{PREFIX}-{key}~, e.g. ~OFX-12345678~, ~CSV-aBcDeFgH==~.

**** CSV digest normalization

For sources without a bank-supplied FITID, hash a canonical string
constructed from raw field values *before* the ~process~ callback runs:

#+begin_example
"$raw_date|$raw_payee|$raw_quantity"
#+end_example

Fields intentionally excluded from the hash:
- *State/status*: pending vs. cleared is volatile; the same transaction must
  get the same ID before and after clearing.
- *Running balance*: changes as subsequent rows clear.
- *Account*: could be renamed without affecting transaction identity.

~ledgertemp4.pl~ hashes the full formatted text (including account lines), which
means any account rename changes the digest and breaks dedup for previously
written template IDs.  Both CSV and TEMP should use the same three-field
canonical form; the date component distinguishes separate occurrences of
recurring transactions.

**** Pending-state flag

Retain the ~!~ suffix for pending transactions.  It is load-bearing: when a
pending transaction is first imported, it writes ~ID: CSV!-abc123~ to the
ledger.  On the next import the bank reports the same transaction as cleared;
~makeid()~ generates ~CSV-abc123~ (no ~!~) → not found in the dedup cache →
falls through to ~balance()~ matching → the pending entry is correctly
transitioned to cleared.  Without ~!~, both the pending and cleared passes
would generate the same key, the cleared version would be deduped, and the
pending transaction would remain in the ledger indefinitely.

**** ~makeid()~ signature change

Remove ~$account~ from the argument list; callers pass the source prefix:

#+begin_src perl
use Digest::MD5 qw(md5_base64);

sub makeid {
    my ($source, $trdat) = @_;
    my $key;
    if ($trdat->{id}) {
        $key = $trdat->{id};
    } elsif ($source eq 'CSV') {
        $key = md5_base64(join('|', $trdat->{date_raw} // '',
                                    $trdat->{payee_raw} // '',
                                    $trdat->{quantity}  // 0));
    } else {
        $key = strftime('%Y/%m/%d', localtime $trdat->{date})
             . '+$' . sprintf('%.02f', $trdat->{quantity});
    }
    $key = $trdat->{salt} . '-' . $key if $trdat->{salt};
    return "$source-$key";
}
#+end_src

Call sites in ~fromStmt~ pass the appropriate prefix:
~makeid('OFX', $stmttrn)~, ~makeid('CSV', $stmttrn)~, ~makeid('PLAID', $stmttrn)~,
~makeid('TELLER', $stmttrn)~.

~date_raw~ and ~payee_raw~ must be captured by ~parsefile~ before the ~process~
callback can overwrite them; they are stashed on ~$csv~ before the callback
is invoked.

**** Dependency

Add ~Digest::MD5~ to ~Ledger.pm~'s import list and to the dependency table
in ~Ledger.pm/CLAUDE.md~.

**** Transition

Existing ledger files carry old-format tags (~M-2026/03/05+$-250.00~).  On
the first import run after the change:

1. ~makeid()~ generates a new-format key; the dedup cache lookup finds the
   old-format key in the ledger → no match → dedup miss.
2. ~addStmtTran~ falls back to ~balance()~ scoring (date + amount + payee).
   ~balance()~ correctly identifies the existing transaction and skips it.
3. After that run, new-format IDs are written alongside the old tags.
   Subsequent runs match on the new IDs cleanly.

*Risk*: transactions that score below the ~balance()~ acceptance threshold
will be treated as new imports.  Mitigation: run ~fromStmt~ in dry-run mode
on the first transition import and review the match log before committing
the write.  No bulk migration script is required for the common case.

** FIXED Balance assertions silently dropped in multiple paths         :bug:
:PROPERTIES:
:ID: BUG-011
:END:
:LOGBOOK:
- State "FIXED" from "TODO" [2026-04-20]
:END:

Three separate code paths caused balance assertions from ~addStmtBal~ to be
silently dropped without being written to the ledger file.

*** Path 1: ~@append~ empty (all transactions deduplicated)

~update_file~ only registered the ~ofxpos~ sentinel when ~@append~ was
non-empty.  If every incoming statement transaction matched an existing ID
(second import run, or a statement with only already-cleared transactions),
~@append~ was empty, the sentinel was never inserted, and balance assertions
were never written.

Fix: pre-compute ~@balance_entries~ and widen the condition:

#+begin_src perl
my @balance_entries = sort { $a->{date} <=> $b->{date} }
                          map { values %$_ } (values %{$self->{balance}});
if ($ofx && (@append || @balance_entries)) {
    $posmap{$self->{ofxpos}} //= -1;
}
#+end_src

*** Path 2: ~ofxpos~ blocked by an in-place edit

When a pending transaction at ~ofxpos~ was cleared in-place (e.g. a two-sided
transfer where the match was found via the ledgerCSV queue), its ~bpos~
occupied ~posmap~ before the sentinel could be inserted via ~//=~.  The
~elsif ($ofx)~ block was never reached and balance assertions were dropped.

Fix: after writing any in-place edit at ~ofxpos~, immediately append balance
entries if they have not yet been written:

#+begin_src perl
if ($ofx && $pos == $self->{ofxpos} && !$balance_written && @balance_entries) {
    print $writeh "\n; ".localtime."\n\n";
    print $writeh join("\n", map { $_->toString } @balance_entries)."\n\n";
    $balance_written = 1;
}
#+end_src

A ~$balance_written~ flag prevents double-writing; the end-of-file block uses
the same flag to catch any remaining unwritten entries.

*** Path 3: ~update()~ skips ~update_file~ when ~@edit~ is empty

When all incoming transactions were deduplicated, ~@edit~ was empty.
~%files~ was built entirely from ~@edit~, so ~update_file~ was never called at
all — balance assertions in ~$self->{balance}~ were never written regardless
of the ~update_file~ fixes above.

Fix: unconditionally add ~ofxfile~ to ~%files~ when balance entries exist:

#+begin_src perl
$files{$self->{ofxfile}} = 1
    if $self->{ofxfile} && %{$self->{balance}};
#+end_src

*** Regression test

~t/test_bug011.pl~ uses a fixture with one already-cleared transaction
(~ID: C-CHK-001~) and a CSV that re-imports the same transaction plus a
balance assertion.  All three paths are exercised: the transaction is
deduplicated (~@append~ empty), ~@edit~ is empty, and the balance assertion
must still appear in the resulting file.

** TODO Removed holds left as orphaned pending transactions            :bug:
:PROPERTIES:
:ID: BUG-013
:END:

Vendors occasionally place holds on cards that appear as pending
transactions in Plaid (and possibly Teller).  When the hold is
released without posting, the transaction simply disappears from
subsequent statement fetches.  The importer has no deletion event to
process — it only sees new or updated transactions — so the orphaned
pending entry remains in the ledger until manually removed.

This is distinct from BUG-005, where a pending transaction /posts/ but
its ID changes between the pending and posted states, causing a match
failure.  Here the transaction is never posted at all; the desired
outcome is deletion, not clearing.

*** Root cause

~makeid~ appends ~!~ to the key for pending transactions, making them
distinct in the ~{id}~ cache from any posted counterpart.  This is
correct, but the cache only prevents re-importing a transaction already
seen — it has no mechanism to detect that a previously-imported pending
transaction is no longer present in the current statement.

*** Fix

After all transactions in a statement have been processed, perform an
absence scan:

1. Collect the set of ~makeid~ keys for every pending transaction seen
   in this statement run (keys ending in ~!-...~ for the current
   account).
2. Retrieve all existing pending transactions in the ledger for the
   same account (~getTransactions('pending')~ filtered by account
   posting).
3. For each existing pending transaction whose ~ID:~ tag key is /not/
   in the seen set, schedule it for deletion (i.e. in-place overwrite
   with empty content, analogous to how cleared transactions overwrite
   pending ones).

This scan should be opt-in or source-aware: it is only correct when
the statement provides a /complete/ snapshot of current pending
transactions, not just new ones.  Plaid satisfies this (it always
returns all current pending transactions for an account).  OFX/QFX
provides a date-bounded window and may not include all pending
transactions; CSV almost certainly does not.  A ~pending_snapshot =>
1~ option to ~fromStmt~ (or inferred from the ~.json~ extension) would
gate the scan.

** FIXED Date comment written even when no transactions are appended  :bug:
:PROPERTIES:
:ID: BUG-014
:END:
:LOGBOOK:
- State "FIXED" from "TODO" [2026-04-20]
:END:

~update_file~ writes a ~; <localtime>~ separator comment in three places.
Two of those places fire even when ~@append~ is empty — i.e. when only
balance assertions (no transactions) are being written at ~ofxpos~:

1. The ~elsif ($ofx)~ block (processes the ~-1~ sentinel at ~ofxpos~) writes
   the comment unconditionally before printing ~@balance_entries~.
2. The EOF block writes the comment when ~@remaining_bal~ is non-empty,

The comment is intended to mark a batch of newly-imported transactions.

uncleared section that starts at ~ofxpos~ — and can misplace balance
assertions relative to pending transactions, causing balance errors.

Make the ~; <localtime>~ line conditional on ~@cleared || @uncleared~
being non-empty in both blocks:

#+begin_src perl
# elsif ($ofx) block:
print $writeh "\n; ".localtime."\n\n" if @cleared || @uncleared;

# EOF block:
print $writeh '; '.localtime."\n\n" if @cleared || @uncleared;
#+end_src

The balance entries are still written; they just don't get a spurious
date comment when no transactions accompany them.

** DONE gentable / finddest reliability concerns                   :feature:
:PROPERTIES:
:ID: FR-002
:END:
:LOGBOOK:
- State "DONE" from "TODO" [2026-04-17]
:END:

Replaced the frequency-table approach with a proper Naive Bayes text
classifier.  ~gentable~ now tokenises payee strings and builds per-source-account
prior and token-likelihood tables; ~finddest~ computes log-probabilities with
Laplace smoothing and returns a softmax confidence score.  Training data
filtering and the confidence threshold are addressed as part of this change
(see notes on each sub-issue below).

~gentable~ builds the frequency table used by ~finddest~ for auto-categorisation.
Several aspects are questionable:

*** FIXED Trains on all transactions without filtering

Every transaction in the ledger contributes to the table, including:
- Equity and transfer postings (~Equity:Transfers:X~, ~Equity:Opening Balances~)
- Balance assertion pseudo-transactions
- Auto-categorised transactions the library was uncertain about

These pollute the signal.  Note that pending/uncleared transactions should
*not* be excluded wholesale — most are manually entered and are likely correct.
The reliable signal is the posting note: postings with an ~INFO:~ tag were
auto-categorised with low confidence and should be excluded; those without
~INFO:~ were manually verified and are good training data.

The table should exclude:
- Equity and transfer account postings
- Balance assertion transactions
- Any posting whose note contains ~INFO:~

*** TODO ~getPosting(-1)~ as destination is unreliable

The destination account is taken from the *last* posting.  For a standard
two-posting transaction this works, but for transactions with 3+ postings (or
balance assertions attached) the last posting may not be the meaningful
category.

*** FIXED The bracket heuristic is approximate

Amount magnitude is encoded as ~length(sprintf('%.2f', abs($amount)))~:

| Amount range | Bracket |
|--------------+---------|
| $1.00–$9.99  |       4 |
| $10.00–$99.99|       5 |
| $100–$999.99 |       6 |

This creates cliff-edges: $9.99 and $10.00 land in different buckets despite
being nearly the same amount.  It is a reasonable approximation but worth
knowing about when debugging unexpected categorisation.

*** FIXED The ~@@account default@@~ fallback truncates to two account levels

When no payee match is found, ~gentable~ falls back to a bucket keyed on the
first two account levels (~join(':', (split(/:/, $dest))[0,1])~), e.g.
~Expenses:Food:Dining~ → ~Expenses:Food~.  This truncated account may not exist
as a real ledger account, and the fallback wins even when the truncation loses
meaningful specificity.

*** TODO No recency weighting

All historical transactions count equally.  If a payee's category is corrected,
old training data continues to dilute the new signal until enough new
transactions accumulate to overcome it (~finddest~ requires >12 samples at
>99.99% agreement for high-confidence results).

** DONE Payee description cache: variable suffixes cause key explosion  :feature:
:PROPERTIES:
:ID: FR-007
:END:

Resolved by FR-009 (~_token_lookup~).  A desc key of ~"blue bottle"~ now matches
~"SQ *BLUE BOTTLE 123456"~ via token overlap — no wildcard syntax or
normalization heuristic required.  The cache may still accumulate exact-match
entries on first sight, but ~_token_lookup~ makes them unnecessary going forward.



Raw payee strings from bank statements contain variable suffixes that change
per transaction, so ~desc.yaml~ accumulates a separate entry for each variant
even when they all map to the same canonical name:

#+begin_example
AMAZON.COM*A1B2C3D4 AMZN.COM/BILL: Amazon
AMAZON.COM*B5C6D7E8 AMZN.COM/BILL: Amazon
SQ *BLUE BOTTLE 123456: Blue Bottle Coffee
SQ *BLUE BOTTLE 789012: Blue Bottle Coffee
#+end_example

The cache grows unboundedly, edits are futile for the next random suffix, and
there is no way to pre-populate it for new merchants.

~desc~ serves two roles in ~addStmtTran~:
1. Maps raw payee → canonical name used as a handler lookup key (line ~223~)
2. Maps raw payee → canonical name used as the transaction payee (line ~234~)

Both are exact hash lookups today.

*** Proposed solution A: prefix wildcard keys

Allow keys ending in ~*~ to match any payee beginning with that prefix.
The user writes:

#+begin_example
AMAZON.COM*: Amazon
SQ *: Square
#+end_example

A helper ~_desc_lookup($payee)~ does an exact lookup first, then scans for
~*~-suffix keys:

#+begin_src perl
sub _desc_lookup {
    my ($self, $payee) = @_;
    return $self->{desc}{$payee} if exists $self->{desc}{$payee};
    for my $key (grep { /\*$/ } keys %{$self->{desc}}) {
        my $prefix = substr($key, 0, -1);
        return $self->{desc}{$key} if index($payee, $prefix) == 0;
    }
    return undef;
}
#+end_src

Replace the two ~$self->{desc}->{$payee}~ call sites with
~$self->_desc_lookup($payee)~.

Open questions:
- Should a wildcard match write the exact key back into desc for O(1) future
  lookups (cache warm-up), or keep the scan to avoid desc growth?
- Should ~_desc_lookup~ also try ~$cleanpay~ (the ~***~-stripped form) before
  the wildcard scan, or is prefix matching enough?

*** Proposed solution B: normalize before lookup

Strip the variable suffix from the raw payee before any desc lookup, reducing
~AMAZON.COM*A1B2C3D4~ to ~AMAZON.COM*~ automatically.  This avoids user-facing
wildcard syntax but requires a reliable normalization heuristic — difficult
given how varied bank payee strings are in practice.

** TODO Port import pipeline to Go                                 :feature:
:PROPERTIES:
:ID: FR-008
:END:

*** Motivation

- *Single binary distribution*: the Perl library requires ~Text::CSV~,
  ~Date::Parse~, ~JSON~, ~XML::Parser~, and ~YAML::Tiny~ — all stable and
  available in OS package managers, but still friction on a new machine.  A Go
  binary has no runtime dependencies beyond the ~ledger~ CLI itself.
- *Speed*: Go's compiled execution and native concurrency would make the import
  pipeline meaningfully faster, particularly for large statement files or bulk
  imports across many accounts.
- *Unification*: ledger-web already re-implements byte-position tracking and
  in-place editing (~updateFileEdit~).  A Go port would allow that logic to be
  shared rather than maintained in two places.

*** Remaining hard dependency and the real performance bottleneck

Both the Perl and Go versions shell out to ~ledger csv~ to read existing
transactions.  On a real ledger file this takes ~5 seconds (~44k output lines);
everything the Perl library does on top of that is noise.  A Go port that
retains the subprocess call would have the same baseline — the language change
alone would not meaningfully improve speed.

The only way to move that number is to replace ~ledger csv~ with a native
ledger-file parser.  That is a substantially larger undertaking (transaction
syntax, ~!include~, commodities, virtual accounts) and is out of scope for this
work item.  The existing YAML object cache was added to avoid paying the
~ledger csv~ cost on repeat runs, though it is of dubious value now that
ledger-web no longer uses Ledger.pm.

*** Handler strategy

Perl handlers are code refs defined inline in per-user import scripts, giving
full scripting flexibility.  In a compiled Go binary, the equivalent options
are:

1. *YAML/TOML config* for the common case: ~payee → account~ mappings and
   transfer routing rules.  Covers ~80%+ of real-world handlers.

2. *External handler scripts via JSON stdio* for custom logic: the binary
   serialises each unmatched transaction to JSON, pipes it to a user-supplied
   script (any language), and reads back the modified transaction.  Simple
   cases can be handled in a shell script; complex cases in whatever language
   the user prefers.  No embedded interpreter needed.

The recommended approach is to combine both: YAML config as the primary
interface, with an optional ~--handler~ flag pointing to an executable for
edge cases.

*** Scope

- OFX, CSV, and JSON (Plaid/Teller) parsers
- Heuristic matching (~distance~, ~checkpending~, ~balance~)
- Transfer matching
- In-place file editing (reuse or replace ledger-web's ~updateFileEdit~)
- Payee description cache (~desc.yaml~)
- YAML config + JSON stdio handler protocol

** DONE Replace desc.yaml and handler lookup with token-based matching :feature:
:PROPERTIES:
:ID: FR-009
:PRIORITY: high
:END:

~desc.yaml~ and the handler map both use exact raw payee strings as keys.
This requires a separate entry per bank-string variant (see payee cache key
explosion issue), and means handlers must enumerate every spelling a bank
might use for a given merchant.

A token-based lookup would treat the keys as sets of significant words
rather than exact strings.  The lookup would tokenize the incoming raw
payee (using ~_tokenize~), then find the handler or canonical name whose
key tokens best overlap with the incoming tokens.

#+begin_example
handler key: "blue bottle"   → tokens: [blue, bottle]
raw payee:   "SQ *BLUE BOTTLE 123456"  → tokens: [sq, blue, bottle]
recall: 2/2 = 100%  → matched
#+end_example

This would subsume both proposed solutions to the key explosion problem
(prefix wildcards and normalization heuristics) with a single, more
principled mechanism.  The Naive Bayes classifier already builds a
compatible token vocabulary; the handler lookup could reuse ~_tokenize~
directly.

Key open questions:
- Threshold for a "match" (to avoid spurious overlaps on common words like

** TODO Integrate ledgertemp2.pl as Ledger::Template               :feature:
:PROPERTIES:
:ID: FR-012
:END:

~ledgertemp2.pl~ (and ~ledgertemp3.pl~) append directly to the ledger file with
~>>~, bypassing Ledger.pm's ~ofxpos~ insertion point.  This causes file
degradation: cleared template transactions land after the uncleared section,
which subsequent ~stmt2ledger.pl~ runs interleave with new transactions.

The fix is to route template output through Ledger.pm's write path.  The core
challenge is that template entries are raw ledger transaction strings — the
frequency/date header is stripped but the body is never parsed.

*** Template file format

Each paragraph in a ~.tpl~ file consists of a frequency header on the first
line, followed by a raw ledger transaction body with an ~ID:~ placeholder:

#+begin_example
W3
Transit Authority
    Expenses:Transportation:Transit     $90.00
    Assets:Checking                    ;ID:
#+end_example

The frequency code (~W~=weekly, ~B~=biweekly, ~M~=monthly) and day offset are
parsed; everything else is treated as an opaque string.  The script generates
occurrence dates, computes an ID key from the body, substitutes the date and
key into the string, and prints.

*** Three implementation paths

**** Path A: Parse raw ledger text → Transaction objects (recommended long-term)

Add ~Ledger::Template~ with a ~parsefile~ function dispatched from ~fromStmt~
(~.tpl~ extension).  Parse the raw transaction body into a ~Ledger::Transaction~
object: payee line, state marker, and postings.  Call ~addTransaction~ directly,
bypassing ~addStmtTran~ (templates are already fully categorised).

Pros:
- Template files stay in human-readable ledger format; existing ~.tpl~ files
  require only minor header adjustments.
- Full ~Ledger::Transaction~ objects: ID dedup via ~$self->{id}~, proper
  ~ofxpos~ insertion, testable.
- ID generation unified under ~Ledger::makeid~; format mismatch eliminated.

Cons:
- Requires writing a ledger transaction text parser — something Ledger.pm has
  never needed (it reads via ~ledger csv~, not directly from ~.dat~ files).
  The needed subset is narrow (payee, state, account+amount postings) but is a
  new surface to keep correct.
- Ledger idioms not used in templates today (commodities, virtual accounts)
  will cause parser failures if they appear later.

**** Path B: New structured template format (YAML)

Replace the raw-string body with structured YAML:

#+begin_src yaml
- freq: W
  day: 3
  payee: Transit Authority
  postings:
    - account: Expenses:Transportation:Transit
      amount: "90.00"
    - account: Assets:Checking
#+end_src

Pros:
- No text parser needed; maps directly to ~Ledger::Transaction~ fields.
- ~YAML::Tiny~ is already a dependency.

Cons:
- Breaking format change; existing ~.tpl~ files must be migrated.
- More verbose; loses the ledger-native syntax users already know.

Only worthwhile if templates are ever generated programmatically.

**** Path C: Keep strings, fix only the insertion point (recommended short-term)

Keep raw transaction strings.  Add a ~fromTemplate~ method that generates
occurrence dates, computes the ID key (see below), checks ~$self->{id}~, and
appends the raw string at ~ofxpos~ via ~update_file~.  Solves file degradation
without a transaction parser.

Cons:
- Not zero parsing: computing the ID key requires extracting the account name
  and amount from the first posting line (one regex, but still parsing).
- Format coupling: ~ledgertemp2.pl~'s ~makeid~ produces ~EC-TEMP-2026-04-20+90.00~
  while ~Ledger::makeid~ produces ~EC-2026/04/20+$90.00~ — different date
  separators, different ~$~ handling, different infix.  Previously written
  template IDs are baked into the ledger; changing the format requires either
  a migration or a dual-format lookup in ~$self->{id}~.
- No ~Ledger::Transaction~ object: cannot edit in place later, harder to test.

*** Dedup mechanism

~$self->{id}~ is populated during construction from ~ledger csv~ output:
~CSV.pm~ line 92 sets ~$ledger->{id}->{$csv{key}} = $csv{payee}~ for every
posting that carries an ~ID:~ tag.  Previously written template transactions
are already present in this hash.

For Path A/B, ~addTransaction~ calls ~makeid~ to generate the lookup key and
checks ~$self->{id}~ before writing — the same path used by all other
importers.

For Path C, the caller must reproduce ~ledgertemp2.pl~'s key formula
(~{initials}-TEMP-{YYYY-MM-DD}{±amount}~) from the raw string in order to
perform the same lookup.

*** Relationship to ledgertemp3.pl

~ledgertemp3.pl~ takes explicit begin/end dates and has no ID deduplication by
design — it regenerates the full range on every run.  It writes to a separate
planning register (~check.ldg~), not the main cleared ledger.  If ~check.ldg~
is not part of the main ledger managed by Ledger.pm, there is no file
degradation problem and no pressure to integrate it.  Defer unless ~check.ldg~
is added to the main ~!include~ tree.

** DONE Track separate insertion files for cleared, pending, and uncleared transactions :feature:
:PROPERTIES:
:ID: FR-014
:END:
:LOGBOOK:
- State "DONE" from "TODO" [2026-04-20]
:END:

Currently all imported transactions are inserted at a single ~ofxpos~
— the byte position of the first non-cleared transaction in ~ofxfile~
(or EOF).  ~ofxfile~ itself is fixed to the file containing the last
cleared transaction.  This means every new transaction, regardless of
state, lands at the same point, which breaks when the ledger is split
across files by state.

*** Goal

Support a ledger organised as:

#+begin_example
!include cleared.ldg
!include pending.ldg
!include uncleared.ldg
#+end_example

Each new transaction should be appended to the file that matches its
state, so the three files stay coherent over time.

*** Design

Replace the single ~ofxfile~/~ofxpos~ pair with three independent
~(file, pos)~ pairs, one per state.

**** Insertion point logic (single-file and multi-file)

For each state, the insertion point is the ~bpos~ of the first
transaction whose state sorts /after/ the target state in the sequence
cleared → pending → uncleared, within the target file.  If no such
transaction exists, the insertion point is EOF of the target file.
This keeps each section contiguous regardless of whether the sections
live in one file or three.

Concretely, within a given file:

| Inserting   | Insertion point                                          |
|-------------+----------------------------------------------------------|
| cleared     | ~bpos~ of first pending or uncleared txn; else EOF       |
| pending     | ~bpos~ of first uncleared txn; else EOF                  |
| uncleared   | EOF                                                      |

(This generalises the current ~getofxpos~ logic, which already uses
~bpos~ of the first non-cleared transaction as the single insertion
point.)

**** File routing

| State     | Target file                                     | Fallback        |
|-----------+-------------------------------------------------+-----------------|
| cleared   | file of last cleared txn                        | —               |
| pending   | file of last pending txn                        | uncleared file  |
| uncleared | file of last uncleared txn                      | cleared file    |

For single-file setups all three resolve to the same file and the
insertion points within that file keep the sections ordered.  For
multi-file setups (~!include cleared.ldg~ etc.) each state maps to its
own file and the insertion point is EOF of that file (since no
later-state transactions exist there).

**** Detection and fallback

No special ~!include~ scanning is required — file routing falls out
naturally from which file each state's last transaction lives in.  If a
state has no existing transactions the fallback column above applies.
If none of the fallbacks resolve (empty ledger), behaviour is unchanged
from today.

This applies globally — ~fromStmt~, ~scheduleAppend~, and any other
path that calls ~update()~.

*** Implementation

~getofxpos~ replaced by ~getinsertionpoints~, which computes three ~(file,
pos)~ pairs (~cleared_file~/~cleared_pos~, ~pending_file~/~pending_pos~,
~uncleared_file~/~uncleared_pos~).  ~ofxfile~/~ofxpos~ remain as
backward-compatible aliases for the cleared pair.

~addStmtTran~ now calls ~insertionFileFor($state)~ to select the target
file before scheduling an append, rather than always using ~ofxfile~.

~update_file~ replaced the single ~$ofx~/~ofxpos~ sentinel with a
~%sentinels~ hash (~pos => [states]~).  Each sentinel writes only its
own state's ~@append~ transactions; balance entries go at the cleared
sentinel.  Multiple states can share the same sentinel position (e.g.
single-file setup) and are all handled in one pass.

~checkpending~'s pending→cleared branch changed from
~scheduleAppend($candidate->{file})~ to
~scheduleAppend($self->{edit} || $candidate->{file})~ so the moved
transaction targets the cleared file in multi-file setups.

A pre-existing bug in ~ledgerCSV~ was fixed as a prerequisite:
~%(xact.id)~ resets to 1 for each ~!include~ sub-file, so consecutive
rows from different files with the same ID were merged into one
transaction.  The boundary check now uses ~id != $last_id || file ne
$last_file~.

Covered by ~t/test_fr014.pl~ (34 assertions: single-file and
multi-file scenarios).

Fidelity brokerage CSV exports contain transactions from multiple accounts

#+begin_example
Run Date,Account,Account Number,Action,Symbol,Description,Type,Price ($),Quantity,Commission ($),Fees ($),Accrued Interest ($),Amount ($),Settlement Date
#+end_example

Key differences from the CSV sources already supported:

- *Multi-account*: the ~Account~ and ~Account Number~ columns identify which
  brokerage account each row belongs to.  The current ~fromStmt~ model dispatches
  on a single account derived from the filename; a multi-account file requires
  per-row account resolution.
- *Investment actions*: ~Action~ (e.g. ~Buy~, ~Sell~, ~Dividend~), ~Symbol~,
  ~Quantity~, and ~Price~ map naturally to the commodity/cost/quantity fields
  already used by ~Ledger::OFX~ investment transactions.
- *No balance column*: the file does not provide running or end-of-period
  balance data, so no balance assertion can be generated.

*** Account naming — existing patterns to reuse

OFX.pm already handles both forms of investment account-name augmentation:

1. *Commodity suffix* (~OFX::stop~, line 160):
   #+begin_src perl
   $post->{account}.=":$commodity"
   #+end_src
   Called after all transactions are parsed, once the ticker is known from
   ~<SECINFO>~.  Appends the resolved ticker symbol (e.g. ~:AAPL~) to the
   posting account.  The Fidelity CSV provides ~Symbol~ inline on each row, so
   the suffix can be appended immediately in the ~process~ callback rather than
   deferred to a ~stop()~-equivalent.

2. *401k source suffix* (~OFX::inv~, line 223):
   #+begin_src perl
   $posting->{account}.=':'.$arg->{inv401ksource}
   #+end_src
   Appends the OFX ~INVTRAN/INV401KSOURCE~ value (typically ~PRETAX~, ~MATCH~,
   ~AFTERTAX~, ~ROLLOVER~) to the posting account after the callback returns.
   The Fidelity ~Type~ column carries equivalent information for 401k accounts;
   the ~process~ callback should append it to ~$tran{account}~ when non-empty.

Plaid's ~getaccounts~ (Plaid.pm lines 44–55) builds the same ~:TICKER~ suffix at
account-object construction time rather than post-callback; either approach works
since ~addStmtTran~ reads ~$stmttrn->{account}~ before creating the posting.

*** Implementation sketch

- Parse the file in a single pass with ~Ledger::CSV::parsefile~.  The ~process~
  callback (already supported) handles per-row logic:
  - Resolve ~Account Number~ to a full ledger account via ~accounttab~ or a
    YAML mapping; set ~$tran{account}~.
  - Append ~:$symbol~ from the ~Symbol~ column when non-empty (commodity
    positions), mirroring ~OFX::stop~.
  - Append ~:$type~ from the ~Type~ column when it indicates a 401k source
    (~PRETAX~, ~MATCH~, etc.), mirroring ~OFX::inv~ line 223.
  - Map ~Action~ to ~quantity~/~cost~/~commodity~: ~Buy~ → units + cost basis;
    ~Sell~ → negative units; ~Dividend~ → cash amount; etc.
- ~addStmtTran~ requires no changes: it already uses ~$stmttrn->{account}~ and
  handles commodity/cost fields.
- No balance assertion is generated (the file has no position totals).

*** Implementation

Uses ~header_map~ (FR-016) for column-name-driven field mapping and a ~process~
callback closure for all Fidelity-specific logic.  No changes to ~CSV.pm~ or
~Posting.pm~.

Key column mappings: ~Run Date~ → ~date~, ~Account Number~ → ~account_number~,
~Action~ → ~action~, ~Symbol~ → ~symbol~, ~Description~ → ~payee~, ~Type~ →
~contrib_type~, ~Price ($)~ → ~price_col~, ~Quantity~ → ~shares~, ~Amount ($)~
→ ~quantity~.

The ~process~ callback:
- Resolves ~account_number~ via a user-supplied ~%account_map~ closure
- For buy/sell actions: sets ~cost = abs(Amount)~ (falls back to ~Price × Quantity~
  if Amount is blank), ~quantity = shares~, ~commodity = Symbol~, appends

- Appends ~:Type~ to account when ~contrib_type~ is set (401k ~PRETAX~/~MATCH~/etc.)
- Cash actions (dividend, interest): passes ~quantity~ (Amount) through unchanged

Output format: ~2 FXAIX @@ $250.00~ — total cost via existing ~cost~ field in
~Posting.pm~.

A ready-to-use import script is provided at ~scripts/fidelity.pl~.  It wraps

**** Option A: end-of-statement BAL (existing pattern)

the combined amount + assertion form.

#+begin_src perl
# constructor: @{$self}{qw(account quantity commodity cost note assert)}=@_;
# toString, after sprintf('$%0.2f', $self->{quantity}):
$str .= sprintf(' = $%0.2f', $self->{assert}) if defined $self->{assert};
#+end_src

- ~header_map~ (hashref ~field_name => 'Column Header'~): reads the first

** TODO CSV format auto-detection via header fingerprinting            :feature:
:PROPERTIES:
:ID: FR-019
:END:

The current dispatch path in ~fromStmt~ requires the CSV filename to follow
the ~AccountPrefix-date.csv~ convention so that the stripped prefix matches
a key in the ~%csv~ config map:

#+begin_src perl
my $account = $stmt;
$account =~ s/-.*//;       # "Checking-2026-03.csv" → "Checking"
$account =~ s!.*/!!;
$account =~ s/\..*//;
# ...
&Ledger::CSV::parsefile($stmt, $csv->{$account}, $callback);
#+end_src

This conflates two independent roles: the config-dispatch key and the
default ledger account name.  ~#LedgerName:~ (FR-016) decouples the
account name, but the config key still depends on the filename.

*** Goal

Allow a CSV file to be processed correctly even when its name does not
embed the config key — for example, a download saved as
~export-2026-03.csv~ or a file whose institution name differs from the
config key.

*** Proposed approach: fingerprint key in config entries

Add an optional ~fingerprint~ key to each ~%csv~ config entry.  Its value
is the exact header line (or a regex) expected for that format.
~parsefile~ reads the first non-directive line of the file; if it matches
any registered fingerprint, that config entry is used regardless of the
filename prefix:

#+begin_src perl
# Caller config:
Fidelity => {
    fingerprint   => qr/^Run Date,Account,Account Number,Action/,
    header_map    => { ... },
    process       => sub { ... },
},
HSA => {
    fingerprint   => qr/^Transaction Date,Transaction Type,Claimant/,
    header_map    => { ... },
    running_balance => 'available_balance',
    process       => sub { ... },
},
#+end_src

Detection logic added to ~fromStmt~ (or ~parsefile~):

1. If ~$csv->{$account}~ exists (filename-prefix match), use it as today.
2. Otherwise, peek at the first header line of the file and scan all
   registered config entries for a ~fingerprint~ match.
3. If a fingerprint matches, use that config entry.
4. If no match, die or warn with a clear message (unknown format).

The peek must be done before the ~parsefile~ call.  ~parsefile~ already
handles the ~#LedgerName:~ directive as a leading line, so a two-pass open
or ~seek~ is needed — or ~fromStmt~ can pass the resolved config in.

*** Relationship to per-row account resolution

For multi-account files (FR-015), the account varies per row.  Once the
format is fingerprinted, ~fromStmt~ passes a catch-all config (e.g. keyed
on ~'*'~) and the ~process~ callback resolves the per-row ~Account Number~
via ~accounttab~ or a user-supplied map.  Fingerprinting makes this work
even when the download filename contains no useful prefix.

*** Tradeoffs

- A fingerprint registry must be maintained by the caller; genuinely
  unknown CSV formats still need explicit config.
- If two institutions share an identical header (unlikely but possible),
  the fingerprint is ambiguous.  The filename prefix remains a valid
  tie-breaker when present.
- The ~seek~-back after peeking the header adds a small complexity cost to
  ~parsefile~ or ~fromStmt~; the existing ~#LedgerName:~ seek is a
  precedent.

Institution-specific CSV parsing logic is currently written as inline
closures in import scripts (e.g. the Fidelity and HSA configs in
~t/test_fr015.pl~ and ~t/test_fr016.pl~).  This means the same
~header_map~/~process~ block must be duplicated in every script that
imports from that institution.

*** Goal

Extract each institution's format config into a named
~Ledger::CSV::Institution~ module.  Each module declares its own
fingerprint regex so that ~fromStmt~ (via the FR-019 dispatch path) can
select it automatically when the CSV header matches.  Account names are
always caller-supplied — never hardcoded in the module.

*** Module interface

Each module exports two class methods:

- ~fingerprint()~ — returns a compiled regex that uniquely matches the
  institution's CSV header line.
- ~config(%opts)~ — returns a hashref ready to pass as ~$args~ to
  ~parsefile~.  ~%opts~ threads caller data (e.g. ~account_map~) into
  the returned closures without exposing internal field names.

#+begin_src perl
package Ledger::CSV::Fidelity;

sub fingerprint { qr/^Run Date,Account,Account Number,Action/ }

sub config {
    my ($class, %opts) = @_;
    my $account_map = $opts{account_map} // {};
    my $buy_re  = qr/bought|buy|contribution|reinvestment/i;
    my $sell_re = qr/sold|sell|redemption/i;
    return {
        header_map => {
            date           => 'Run Date',
            account_number => 'Account Number',
            action         => 'Action',
            symbol         => 'Symbol',
            payee          => 'Description',
            contrib_type   => 'Type',
            price_col      => 'Price ($)',
            shares         => 'Quantity',
            quantity       => 'Amount ($)',
        },
        process => sub {
            my $csv = shift;
            my $base = $account_map->{$csv->{account_number}}
                       // $csv->{account_number};
            my $action = $csv->{action} // '';
            if ($action =~ $buy_re || $action =~ $sell_re) {
                my $amt = $csv->{quantity} + 0;
                $amt = $csv->{price_col} * abs($csv->{shares})
                    if !$amt && $csv->{price_col};
                $csv->{cost}      = abs($amt);
                $csv->{quantity}  = $csv->{shares} + 0;
                $csv->{quantity}  = -$csv->{quantity} if $action =~ $sell_re;
                $csv->{commodity} = $csv->{symbol};
                $base .= ":$csv->{symbol}" if ($csv->{symbol}//'') =~ /\S/;
            }
            $base .= ":$csv->{contrib_type}"
                if ($csv->{contrib_type}//'') =~ /\S/;
            $csv->{account} = $base;
            $csv->{state}   = 'cleared';
        },
    };
}
1;
#+end_src

*** Relationship to FR-019

FR-019 adds fingerprint dispatch logic to ~fromStmt~: when the filename
prefix does not match a config key, it peeks the CSV header and scans
registered fingerprints.  FR-020 provides the modules that supply those
fingerprints.  The two are complementary but independent — FR-020 modules
can be used before FR-019 is implemented by calling ~config()~ explicitly
in the import script:

#+begin_src perl
use Ledger::CSV::Fidelity;
my %csv = (
    Fidelity => Ledger::CSV::Fidelity->config(account_map => \%ACCOUNTS),
);
$ledger->fromStmt($file, {}, \%csv);
#+end_src

*** Relationship to FR-015 and FR-016

The Fidelity inline config in ~t/test_fr015.pl~ and the HSA inline config
in ~t/test_fr016.pl~ are the reference implementations that move into
~Ledger/CSV/Fidelity.pm~ and ~Ledger/CSV/HSA.pm~ respectively.  No logic

*** Starting modules

**** ~Ledger::CSV::Fidelity~

Fingerprint:
#+begin_example
Run Date,Account,Account Number,Action,Symbol,Description,Type,Price ($),Quantity,...
#+end_example
~qr/^Run Date,Account,Account Number,Action/~

Migrated from FR-015.  ~config(account_map => \%map)~ — caller supplies an
~account_number → ledger_account~ map for per-row account resolution.
Handles Buy/Sell with commodity+cost, Dividend as cash (no symbol suffix),
and 401k contributions with ~contrib_type~ (~PRETAX~/~MATCH~/etc.) appended
to the account.

**** ~Ledger::CSV::Coinbase~

Fingerprint:
#+begin_example
Timestamp,Transaction Type,Asset,Quantity Transacted,Price Currency,Price at Transaction,Subtotal,Total (inclusive of fees and/or spread),Fees and/or Spread,Notes
#+end_example
~qr/^Timestamp,Transaction Type,Asset,Quantity Transacted/~

Coinbase Advanced Trade CSV export (single account per download).
Account name supplied via ~#LedgerName:~ directive in the file —
no ~account_map~ needed.

Column mapping:
- ~Timestamp~ → ~date~
- ~Asset~ → ~commodity~
- ~Quantity Transacted~ → ~quantity~
- ~Total (inclusive of fees...)~ → ~cost~ (Buy/Sell only, absolute value)
- ~Transaction Type~: ~Buy~/~Sell~ → commodity posting with cost; ~Receive~/~Send~
  → treat as transfer (set ~account~ to ~Equity:Transfers:$asset~); ~Rewards Income~
  → pass through as income posting with no cost
- No running balance column; no balance assertion emitted.

**** ~Ledger::CSV::HSA~

Migrated from FR-016.  Fingerprint:
~qr/^Transaction Date,Transaction Type,Claimant,Description/~

Account via ~#LedgerName:~ directive.  ~running_balance => 'available_balance'~
produces an end-of-statement BAL assertion from the last cleared row.
~Status~ column maps to cleared/pending state.

*** ~Ledger::CSV::detect($header_line)~

A new utility function in ~Ledger/CSV.pm~ that accepts a raw header line
and returns the first module whose ~fingerprint()~ matches, or ~undef~.
FR-019's ~fromStmt~ fallback calls this after the filename-prefix lookup
fails.  The registry is a static list in ~CSV.pm~ (not a directory scan)
to avoid introducing ~File::Find~:

#+begin_src perl
my @KNOWN_MODULES = qw(
    Ledger::CSV::Fidelity
    Ledger::CSV::Coinbase
    Ledger::CSV::HSA
);

sub detect {
    my $header = shift;
    for my $mod (@KNOWN_MODULES) {
        (my $file = $mod) =~ s!::!/!g;
        require "$file.pm";
        return $mod if $header =~ $mod->fingerprint();
    }
    return undef;
}
#+end_src

~fromStmt~ calls ~detect~, then passes ~$mod->config(%caller_opts)~ as the
resolved config.  ~%caller_opts~ is taken from a new optional fourth
argument to ~fromStmt~ (or from a key in the existing ~%handlers~).

*** Process callback composition

When a caller needs to augment a module's ~process~ (e.g. to set a custom
note or override one field), they wrap the returned config, not the module
internals:

#+begin_src perl
my $config = Ledger::CSV::Fidelity->config(account_map => \%map);
my $inner  = $config->{process};
$config->{process} = sub {
    my $csv = shift;
    $inner->($csv);
    $csv->{note} = "fidelity import $date";
};
#+end_src

Modules do not call a caller-supplied hook internally; composition is the
caller's responsibility.  This keeps module ~process~ closures testable in
isolation.

*** Tests

Each new module needs a test file:

- ~t/test_fr020_fidelity.pl~ — thin wrapper: calls
  ~Ledger::CSV::Fidelity->config(account_map => ...)~ and re-runs the
  FR-015 assertions.  Proves the migration is behaviour-identical.
- ~t/test_fr020_coinbase.pl~ — new fixture ~t/fr020_coinbase.csv~
  (with ~#LedgerName:~ prefix) covering Buy, Sell, and Rewards Income rows.
- ~t/test_fr020_hsa.pl~ — thin wrapper over FR-016 fixture.

Existing ~test_fr015.pl~ and ~test_fr016.pl~ continue to pass unchanged
(inline config still valid; modules are additive).

*** Tradeoffs

- Callers must still supply account names explicitly; no defaults are
  baked into modules.
- A static registry in ~CSV.pm~ must be updated when a new module is
  added.  This is preferable to auto-discovery via ~File::Find~ given the
  small, bounded module set.
- Callback composition is caller-side.  Module ~process~ closures should
  be kept minimal and focused so wrapping them is straightforward.
- If two institutions share an identical header prefix, the fingerprint
  is ambiguous; the static registry order acts as the tiebreaker.
