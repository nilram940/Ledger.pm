** TODO Account identification via filename is fragile              :bug:
:PROPERTIES:
:ID: BUG-009
:END:

Both importers derive the account name from the statement filename:
~fromStmt~ strips the path, strips everything from the first ~-~, and strips
the extension:

#+begin_src perl
$account =~ s/-.*//;   # "Checking-2026-03.csv" → "Checking"
#+end_src

This string is then used as:
1. A key into the ~%handlers~ map to find the right callback
2. A key into the ~%csv~ config map to find field layout
3. The default account name passed to ~addStmtTran~ / ~addStmtBal~

This means filenames must follow an exact convention, and the short prefix
(e.g. ~Checking~) is used as both a config key and a ledger account name,
conflating two distinct concepts.

*** What the importers already know

- *OFX*: ~<ACCTID>~ and ~<BANKID>~ are parsed into ~$data->{acctid}~ by the
  ~acctid~ handler but are never surfaced to the callback or used for routing.
  The file contains the authoritative account identity.
- *CSV*: Account may appear as a field in the row (~account~ in ~fields~), but
  the config-dispatch key and the default account both still come from the
  filename.
- *Plaid/Teller JSON*: Structured account objects with IDs are available.

*** Prior attempt

~accounttab~ was intended to map 4-digit account-number suffixes to full
ledger account names, but ~getaccount~ was never wired into ~fromStmt~ and
remains a standalone function callers must invoke manually (see BUG-001).

*** Proposed solutions

1. *Explicit account in CSV config*: add an ~account~ key to the per-source
   config hashref.  The filename prefix remains the config dispatch key but is
   no longer used as the ledger account name:
   #+begin_src perl
   Checking => { account => 'Assets:Checking:MyBank',
                 fields  => [qw(date id payee quantity)] }
   #+end_src

2. *OFX routing table*: a YAML config mapping ~BANKID:ACCTID~ → full ledger
   account name.  ~fromStmt~ passes the parsed ~acctid~ to a lookup function
   instead of using the filename.  Subsumes ~accounttab~ with a working
   implementation.

3. *Explicit ~account~ parameter to ~fromStmt~*: simplest possible fix — let
   the caller pass the account name directly and use the filename only as a
   config-dispatch key.

Solutions 1 and 3 are complementary and low-risk; solution 2 is the proper
long-term fix for OFX but requires surfacing ~acctid~ from the OFX parser to
~fromStmt~.

4. *Preprocessor with ~LEDGERNAME:~ injection* (preferred for OFX):
   A standalone script reads the ~<ACCTID>~ from the OFX file, looks it up in a
   mapping (YAML or same ~accounttab~ format), and injects a ~LEDGERNAME:~ line
   into the OFX free-form header before Ledger.pm sees the file:

   #+begin_example
   OFXHEADER:100
   DATA:OFXSGML
   LEDGERNAME:Assets:Checking:MyBank
   ...blank line...
   <OFX>...
   #+end_example

   ~Ledger::OFX::parse~ already splits off the header block (everything before
   the first ~<~) but currently discards it.  A one-line change reads
   ~LEDGERNAME:~ from that block and passes it to the callback, which uses it
   instead of the filename-derived prefix.

   This approach mirrors the existing pattern in ~getplaid3.pl~, where the
   downloader's ~override~ stanza injects ~ledger_name~ into the JSON before
   writing to disk, so ~Ledger::JSON::Plaid~ finds it in the data without any
   special-case logic in ~fromStmt~.  The preprocessor can be a standalone
   script (one per institution, or a generic one driven by a YAML mapping) and
   keeps the injection *outside* Ledger.pm entirely.

** INVESTIGATING Constructor arguments are poorly integrated             :bug:
:PROPERTIES:
:ID: BUG-001
:END:

Several ~Ledger->new~ constructor arguments are accepted but not properly wired
into the library's own logic.  Investigation needed to decide whether to fix
each properly or remove it.

*** ~idtag~

Stored as ~$self->{idtag}~ but never read back.  Four sites hardcode ~'ID:'~
regardless of the option (see separate issue below).  Resolution: either thread
the tag through all call sites, or remove the option and document ~'ID'~ as
fixed.

Deferred to BUG-002, which tracks this in full detail.

*** ~accounttab~ / ~getaccount~

~accounttab~ is loaded into ~$self->{accounts}~ via ~getacctnum~, but:

- ~getaccount~ is a package-level function, not a method; callers must pass
  ~$self->{accounts}~ explicitly as a second argument.
- ~$self->{accounts}~ is never used internally — the library never resolves
  account numbers itself.
- ~getacctnum~ redundantly sets ~$self->{accounts}~ both inside the method and
  via the return-value assignment in ~new()~.

The net effect is that ~accounttab~ loads a table that Ledger.pm never consults;
callers must manually plumb it through to ~getaccount~.  Resolution: either make
~getaccount~ a method (using ~$self->{accounts}~ directly) and wire it into
~fromStmt~/~addStmtTran~ so account numbers are resolved automatically, or
remove ~accounttab~ from the constructor and leave account resolution entirely to
the caller.

*Recommended resolution*: remove ~accounttab~ from the constructor.  Wiring
~getaccount~ into ~addStmtTran~ requires deciding when to resolve (OFX has
~acctid~ in parsed data; CSV may not), which overlaps with the larger
filename-vs-account design work in FR-009.  Leaving resolution to callers is
simpler and honest about what the code actually does.

** TODO idtag option is not consistently honoured                       :bug:
:PROPERTIES:
:ID: BUG-002
:END:

~Ledger->new~ accepts an ~idtag~ constructor parameter (default ~'ID'~) that is
meant to control which ledger tag stores transaction IDs.  The value is stored
in ~$self->{idtag}~ but is never read back; all of the following sites
hardcode the string ~ID:~ instead:

| File | Line | Hardcoded use |
|------+------+---------------|
| ~Ledger.pm~ | 244 | ~"ID: $key"~ — posting note written in ~addStmtTran~ |
| ~Ledger/Posting.pm~ | 30 | ~/ID:\s*(\S+)/~ — ~getid()~ regex |
| ~Ledger/XML.pm~ | 140 | ~/^\s*ID:\s*(\S+)/~ — XML import ID extraction |
| ~Ledger/CSV.pm~ | 110 | ~$csv{note}!~/ID:/~ — transfer detection heuristic |

Two possible resolutions:

1. *Thread the tag through*: pass ~$idtag~ (or a compiled regex) to each site.
   ~getid~ in ~Posting.pm~ is the most impactful since it drives ~distance()~
   ID-based matching.  The ~Ledger~ object is not in scope in ~Posting~ or ~CSV~,
   so the tag would need to be passed explicitly or stored on each posting at
   construction time.

2. *Remove the option*: ~'ID'~ appears to be the only tag in use in practice.
   If no caller has ever needed a different tag, the right fix may be to delete
   the ~idtag~ constructor parameter, hardcode ~'ID'~ openly, and document the
   limitation.  This is simpler and honest about what the code actually does.

** DONE Replace strdist with token-based recall in distance()          :bug:
:PROPERTIES:
:ID: BUG-003
:PRIORITY: high
:END:

~distance()~ uses ~strdist~ to compare payees: it truncates the incoming
statement payee to the length of the existing ledger payee and counts
character mismatches.  This works when the bank string *starts with* the
ledger name, but fails for intermediaries (Square, PayPal, etc.) where the
merchant name does not appear at position 0:

#+begin_example
ledger: "Blue Bottle"          tokens: [blue, bottle]
bank:   "SQ *BLUE BOTTLE 123"  tokens: [sq, blue, bottle]

strdist: substr(bank, 0, 11) = "sq *blue bo"
         vs "blue bottle" → 4 mismatches → distance ≈ 0.36  (wrong)

token recall: [blue, bottle] ∩ [sq, blue, bottle] / 2 = 100%
              → distance contribution 0  (correct)
#+end_example

Proposed fix: replace the ~strdist~ block in ~distance()~ with a token-recall
score — the fraction of the ledger payee's tokens that appear in the statement
payee's tokens.  A recall of 1.0 contributes 0 to ~$dist~; 0.0 contributes the
same weight as the current maximum strdist penalty.  The ~_tokenize~ helper
introduced for the Naive Bayes classifier is already available.

*** TODO Pending (Plaid) transactions do not clear                    :bug:

The automation matches pending transactions by ID, but Plaid's pending ID
either changes or disappears when a transaction posts.  Two failure modes:

1. *ID changes on posting*: the pending transaction has ID ~P-xxx~ and the
   posted transaction arrives with a different ID ~C-xxx~.  The importer
   creates a new cleared transaction, leaving the old pending entry as an
   orphan.

2. *Transaction disappears*: the pending entry is removed from subsequent
   Plaid exports without a posted counterpart (e.g. a cancelled hold, as in
   BUG-013).  The orphan is never matched and remains until manually removed.

Both failure modes produce orphaned pending transactions; BUG-013 (removed
holds) is essentially the same problem scoped to case 2.  A unified fix may
address both:

- *ID-change case*: after the absence scan from BUG-013 identifies a pending
  transaction no longer in the current export, attempt a date+amount match
  against newly-posted cleared transactions before deleting it.  A match
  means the pending entry should be deleted (the cleared version is
  authoritative); no match means the transaction was removed without posting
  and should also be deleted.
- *Both cases* require the ~pending_snapshot~ opt-in from BUG-013 (Plaid
  provides a complete snapshot; OFX/CSV do not).

Possible additional approach: match pending→cleared by date + amount during
normal import, falling back to ID only when an ID match fails.  Risk: date
and amount alone may not be unique enough to avoid false matches.

** TODO makeid collisions and account-name coupling                    :bug:
:PROPERTIES:
:ID: BUG-010
:END:

Generated ID format: ~{initials}[-!]-[salt-]{FITID | date+amount}~

~makeid()~ falls back to ~date+amount~ when no FITID is available.  This can
lead to collisions between distinct transactions that share the same date and
amount.

*** Salt mechanism (partial mitigation)

~makeid~ accepts a ~salt~ field on the transaction hash and inserts it between
the initials prefix and the FITID/fallback:

#+begin_src perl
$id.=$trdat->{salt}.'-' if $trdat->{salt};
#+end_src

Salt is not set by any parser module.  It is set in per-account CSV handler
callbacks for sources whose description field embeds a unique transaction
identifier after asterisks:

#+begin_src perl
if ($csv->{desc} =~ /\*+([^*]+)/) {
    $csv->{salt} = $1;
    $csv->{salt} =~ s/\s+/-/g;
}
#+end_src

For the CSV source this was written for, the extracted string is a stable
per-transaction identifier embedded in the description — effectively a FITID
substitute.  Setting it as ~salt~ rather than ~id~ means ~makeid~ still uses the
date+amount path but prefixes it with the unique token, producing a
collision-free key in practice.  The cleaner fix would be to set ~$csv->{id}~
directly, which would skip the date+amount fallback entirely.

CSV accounts without such a handler remain vulnerable to collision.

*** Account-name coupling

The initials prefix is derived from the last ~:~-separated component of the
account path (e.g. ~Assets:Checking:MyBank~ → ~M~).  Two problems:

1. *Rename breakage*: renaming an account changes the initials, making all
   previously written ~ID:~ tags in the ledger stale — the dedup cache no
   longer matches them on the next import.
2. *Prefix collision*: accounts whose last component shares initials (e.g.
   ~MyBank~ and ~MegaBank~ both give ~M~, or ~Credit Card~ under both ~Assets~
   and ~Liabilities~ both give ~CC~) can collide in the dedup cache across
   accounts.

** FIXED Balance assertions silently dropped in multiple paths         :bug:
:PROPERTIES:
:ID: BUG-011
:END:
:LOGBOOK:
- State "FIXED" from "TODO" [2026-04-20]
:END:

Three separate code paths caused balance assertions from ~addStmtBal~ to be
silently dropped without being written to the ledger file.

*** Path 1: ~@append~ empty (all transactions deduplicated)

~update_file~ only registered the ~ofxpos~ sentinel when ~@append~ was
non-empty.  If every incoming statement transaction matched an existing ID
(second import run, or a statement with only already-cleared transactions),
~@append~ was empty, the sentinel was never inserted, and balance assertions
were never written.

Fix: pre-compute ~@balance_entries~ and widen the condition:

#+begin_src perl
my @balance_entries = sort { $a->{date} <=> $b->{date} }
                          map { values %$_ } (values %{$self->{balance}});
if ($ofx && (@append || @balance_entries)) {
    $posmap{$self->{ofxpos}} //= -1;
}
#+end_src

*** Path 2: ~ofxpos~ blocked by an in-place edit

When a pending transaction at ~ofxpos~ was cleared in-place (e.g. a two-sided
transfer where the match was found via the ledgerCSV queue), its ~bpos~
occupied ~posmap~ before the sentinel could be inserted via ~//=~.  The
~elsif ($ofx)~ block was never reached and balance assertions were dropped.

Fix: after writing any in-place edit at ~ofxpos~, immediately append balance
entries if they have not yet been written:

#+begin_src perl
if ($ofx && $pos == $self->{ofxpos} && !$balance_written && @balance_entries) {
    print $writeh "\n; ".localtime."\n\n";
    print $writeh join("\n", map { $_->toString } @balance_entries)."\n\n";
    $balance_written = 1;
}
#+end_src

A ~$balance_written~ flag prevents double-writing; the end-of-file block uses
the same flag to catch any remaining unwritten entries.

*** Path 3: ~update()~ skips ~update_file~ when ~@edit~ is empty

When all incoming transactions were deduplicated, ~@edit~ was empty.
~%files~ was built entirely from ~@edit~, so ~update_file~ was never called at
all — balance assertions in ~$self->{balance}~ were never written regardless
of the ~update_file~ fixes above.

Fix: unconditionally add ~ofxfile~ to ~%files~ when balance entries exist:

#+begin_src perl
$files{$self->{ofxfile}} = 1
    if $self->{ofxfile} && %{$self->{balance}};
#+end_src

*** Regression test

~t/test_bug011.pl~ uses a fixture with one already-cleared transaction
(~ID: C-CHK-001~) and a CSV that re-imports the same transaction plus a
balance assertion.  All three paths are exercised: the transaction is
deduplicated (~@append~ empty), ~@edit~ is empty, and the balance assertion
must still appear in the resulting file.

** FIXED CSV parser strips minus sign from negative balance assertions  :bug:
:PROPERTIES:
:ID: BUG-012
:END:
:LOGBOOK:
- State "FIXED" from "TODO" [2026-04-20]
:END:

The quantity-cleanup regex in ~Ledger::CSV::parsefile~:

#+begin_src perl
$csv{quantity} =~ s/^.*\$//;
#+end_src

uses a greedy ~.*~ that matches everything up to and including the last ~$~
sign.  For a negative balance assertion like ~= -$812.50~ the substitution
consumed ~-$~, leaving ~812.50~ and silently discarding the sign.

Fix: capture and preserve the optional leading minus:

#+begin_src perl
$csv{quantity} =~ s/^(-?)[^-\d]*\$/$1/;
#+end_src

This strips any currency prefix (including currency symbols and spaces)
while keeping the sign.  Positive amounts are unaffected.

** TODO Removed holds left as orphaned pending transactions            :bug:
:PROPERTIES:
:ID: BUG-013
:END:

Vendors occasionally place holds on cards that appear as pending
transactions in Plaid (and possibly Teller).  When the hold is
released without posting, the transaction simply disappears from
subsequent statement fetches.  The importer has no deletion event to
process — it only sees new or updated transactions — so the orphaned
pending entry remains in the ledger until manually removed.

This is distinct from BUG-005, where a pending transaction /posts/ but
its ID changes between the pending and posted states, causing a match
failure.  Here the transaction is never posted at all; the desired
outcome is deletion, not clearing.

*** Root cause

~makeid~ appends ~!~ to the key for pending transactions, making them
distinct in the ~{id}~ cache from any posted counterpart.  This is
correct, but the cache only prevents re-importing a transaction already
seen — it has no mechanism to detect that a previously-imported pending
transaction is no longer present in the current statement.

*** Fix

After all transactions in a statement have been processed, perform an
absence scan:

1. Collect the set of ~makeid~ keys for every pending transaction seen
   in this statement run (keys ending in ~!-...~ for the current
   account).
2. Retrieve all existing pending transactions in the ledger for the
   same account (~getTransactions('pending')~ filtered by account
   posting).
3. For each existing pending transaction whose ~ID:~ tag key is /not/
   in the seen set, schedule it for deletion (i.e. in-place overwrite
   with empty content, analogous to how cleared transactions overwrite
   pending ones).

This scan should be opt-in or source-aware: it is only correct when
the statement provides a /complete/ snapshot of current pending
transactions, not just new ones.  Plaid satisfies this (it always
returns all current pending transactions for an account).  OFX/QFX
provides a date-bounded window and may not include all pending
transactions; CSV almost certainly does not.  A ~pending_snapshot =>
1~ option to ~fromStmt~ (or inferred from the ~.json~ extension) would
gate the scan.

** FIXED Date comment written even when no transactions are appended  :bug:
:PROPERTIES:
:ID: BUG-014
:END:
:LOGBOOK:
- State "FIXED" from "TODO" [2026-04-20]
:END:

~update_file~ writes a ~; <localtime>~ separator comment in three places.
Two of those places fire even when ~@append~ is empty — i.e. when only
balance assertions (no transactions) are being written at ~ofxpos~:

1. The ~elsif ($ofx)~ block (processes the ~-1~ sentinel at ~ofxpos~) writes
   the comment unconditionally before printing ~@balance_entries~.
2. The EOF block writes the comment when ~@remaining_bal~ is non-empty,
   even if ~@append~ is empty.

The comment is intended to mark a batch of newly-imported transactions.
Writing it without any transactions pollutes the file — specifically the
uncleared section that starts at ~ofxpos~ — and can misplace balance
assertions relative to pending transactions, causing balance errors.

*** Fix

Make the ~; <localtime>~ line conditional on ~@cleared || @uncleared~
being non-empty in both blocks:

#+begin_src perl
# elsif ($ofx) block:
print $writeh "\n; ".localtime."\n\n" if @cleared || @uncleared;

# EOF block:
print $writeh '; '.localtime."\n\n" if @cleared || @uncleared;
#+end_src

The balance entries are still written; they just don't get a spurious
date comment when no transactions accompany them.

*** FIXED Trains on all transactions without filtering

Every transaction in the ledger contributes to the table, including:
- Equity and transfer postings (~Equity:Transfers:X~, ~Equity:Opening Balances~)
- Balance assertion pseudo-transactions
- Auto-categorised transactions the library was uncertain about

These pollute the signal.  Note that pending/uncleared transactions should

- Equity and transfer account postings
- Balance assertion transactions
- Any posting whose note contains ~INFO:~

The destination account is taken from the *last* posting.  For a standard

** DONE Payee description cache: variable suffixes cause key explosion  :feature:
:PROPERTIES:
:ID: FR-007
:END:

Resolved by FR-009 (~_token_lookup~).  A desc key of ~"blue bottle"~ now matches
~"SQ *BLUE BOTTLE 123456"~ via token overlap — no wildcard syntax or
normalization heuristic required.  The cache may still accumulate exact-match
entries on first sight, but ~_token_lookup~ makes them unnecessary going forward.



Raw payee strings from bank statements contain variable suffixes that change
per transaction, so ~desc.yaml~ accumulates a separate entry for each variant
even when they all map to the same canonical name:

#+begin_example
AMAZON.COM*A1B2C3D4 AMZN.COM/BILL: Amazon
AMAZON.COM*B5C6D7E8 AMZN.COM/BILL: Amazon
SQ *BLUE BOTTLE 123456: Blue Bottle Coffee
SQ *BLUE BOTTLE 789012: Blue Bottle Coffee
#+end_example

The cache grows unboundedly, edits are futile for the next random suffix, and
there is no way to pre-populate it for new merchants.

~desc~ serves two roles in ~addStmtTran~:
1. Maps raw payee → canonical name used as a handler lookup key (line ~223~)
2. Maps raw payee → canonical name used as the transaction payee (line ~234~)

Both are exact hash lookups today.

*** Proposed solution A: prefix wildcard keys

Allow keys ending in ~*~ to match any payee beginning with that prefix.
The user writes:

#+begin_example
AMAZON.COM*: Amazon
SQ *: Square
#+end_example

A helper ~_desc_lookup($payee)~ does an exact lookup first, then scans for
~*~-suffix keys:

#+begin_src perl
sub _desc_lookup {
    my ($self, $payee) = @_;
    return $self->{desc}{$payee} if exists $self->{desc}{$payee};
    for my $key (grep { /\*$/ } keys %{$self->{desc}}) {
        my $prefix = substr($key, 0, -1);
        return $self->{desc}{$key} if index($payee, $prefix) == 0;
    }
    return undef;
}
#+end_src

Replace the two ~$self->{desc}->{$payee}~ call sites with
~$self->_desc_lookup($payee)~.

Open questions:
- Should a wildcard match write the exact key back into desc for O(1) future
  lookups (cache warm-up), or keep the scan to avoid desc growth?
- Should ~_desc_lookup~ also try ~$cleanpay~ (the ~***~-stripped form) before
  the wildcard scan, or is prefix matching enough?

*** Proposed solution B: normalize before lookup

Strip the variable suffix from the raw payee before any desc lookup, reducing
~AMAZON.COM*A1B2C3D4~ to ~AMAZON.COM*~ automatically.  This avoids user-facing
wildcard syntax but requires a reliable normalization heuristic — difficult
given how varied bank payee strings are in practice.

** TODO Port import pipeline to Go                                 :feature:
:PROPERTIES:
:ID: FR-008
:END:

*** Motivation

- *Single binary distribution*: the Perl library requires ~Text::CSV~,
  ~Date::Parse~, ~JSON~, ~XML::Parser~, and ~YAML::Tiny~ — all stable and
  available in OS package managers, but still friction on a new machine.  A Go
  binary has no runtime dependencies beyond the ~ledger~ CLI itself.
- *Speed*: Go's compiled execution and native concurrency would make the import
  pipeline meaningfully faster, particularly for large statement files or bulk
  imports across many accounts.
- *Unification*: ledger-web already re-implements byte-position tracking and
  in-place editing (~updateFileEdit~).  A Go port would allow that logic to be
  shared rather than maintained in two places.

*** Remaining hard dependency and the real performance bottleneck

Both the Perl and Go versions shell out to ~ledger csv~ to read existing
transactions.  On a real ledger file this takes ~5 seconds (~44k output lines);
everything the Perl library does on top of that is noise.  A Go port that
retains the subprocess call would have the same baseline — the language change
alone would not meaningfully improve speed.

The only way to move that number is to replace ~ledger csv~ with a native
ledger-file parser.  That is a substantially larger undertaking (transaction
syntax, ~!include~, commodities, virtual accounts) and is out of scope for this
work item.  The existing YAML object cache was added to avoid paying the
~ledger csv~ cost on repeat runs, though it is of dubious value now that
ledger-web no longer uses Ledger.pm.

*** Handler strategy

Perl handlers are code refs defined inline in per-user import scripts, giving
full scripting flexibility.  In a compiled Go binary, the equivalent options
are:

1. *YAML/TOML config* for the common case: ~payee → account~ mappings and
   transfer routing rules.  Covers ~80%+ of real-world handlers.

2. *External handler scripts via JSON stdio* for custom logic: the binary
   serialises each unmatched transaction to JSON, pipes it to a user-supplied
   script (any language), and reads back the modified transaction.  Simple
   cases can be handled in a shell script; complex cases in whatever language
   the user prefers.  No embedded interpreter needed.

The recommended approach is to combine both: YAML config as the primary
interface, with an optional ~--handler~ flag pointing to an executable for
edge cases.

*** Scope

- OFX, CSV, and JSON (Plaid/Teller) parsers
- Heuristic matching (~distance~, ~checkpending~, ~balance~)
- Transfer matching
- In-place file editing (reuse or replace ledger-web's ~updateFileEdit~)
- Payee description cache (~desc.yaml~)
- YAML config + JSON stdio handler protocol

** DONE Replace desc.yaml and handler lookup with token-based matching :feature:

:ID: FR-009
:PRIORITY: high
:END:

~desc.yaml~ and the handler map both use exact raw payee strings as keys.
This requires a separate entry per bank-string variant (see payee cache key

- Threshold for a "match" (to avoid spurious overlaps on common words like

- Whether to weight tokens by inverse frequency (rarer tokens count more)
- How to handle ties between handler keys with equal recall

** TODO Integrate ledgertemp2.pl as Ledger::Template               :feature:
:PROPERTIES:
:ID: FR-012
:END:

~ledgertemp2.pl~ (and ~ledgertemp3.pl~) append directly to the ledger file with
~>>~, bypassing Ledger.pm's ~ofxpos~ insertion point.  This causes file
degradation: cleared template transactions land after the uncleared section,
which subsequent ~stmt2ledger.pl~ runs interleave with new transactions.

The fix is to route template output through Ledger.pm's write path.  The core
challenge is that template entries are raw ledger transaction strings — the
frequency/date header is stripped but the body is never parsed.

*** Template file format

Each paragraph in a ~.tpl~ file consists of a frequency header on the first
line, followed by a raw ledger transaction body with an ~ID:~ placeholder:

#+begin_example
W3
Transit Authority
    Expenses:Transportation:Transit     $90.00
    Assets:Checking                    ;ID:
#+end_example

The frequency code (~W~=weekly, ~B~=biweekly, ~M~=monthly) and day offset are
parsed; everything else is treated as an opaque string.  The script generates
occurrence dates, computes an ID key from the body, substitutes the date and
key into the string, and prints.

*** Three implementation paths

**** Path A: Parse raw ledger text → Transaction objects (recommended long-term)

Add ~Ledger::Template~ with a ~parsefile~ function dispatched from ~fromStmt~
(~.tpl~ extension).  Parse the raw transaction body into a ~Ledger::Transaction~
object: payee line, state marker, and postings.  Call ~addTransaction~ directly,
bypassing ~addStmtTran~ (templates are already fully categorised).

Pros:
- Template files stay in human-readable ledger format; existing ~.tpl~ files
  require only minor header adjustments.
- Full ~Ledger::Transaction~ objects: ID dedup via ~$self->{id}~, proper
  ~ofxpos~ insertion, testable.
- ID generation unified under ~Ledger::makeid~; format mismatch eliminated.

Cons:
- Requires writing a ledger transaction text parser — something Ledger.pm has
  never needed (it reads via ~ledger csv~, not directly from ~.dat~ files).
  The needed subset is narrow (payee, state, account+amount postings) but is a
  new surface to keep correct.
- Ledger idioms not used in templates today (commodities, virtual accounts)
  will cause parser failures if they appear later.

**** Path B: New structured template format (YAML)

Replace the raw-string body with structured YAML:

#+begin_src yaml
- freq: W
  day: 3
  payee: Transit Authority
  postings:
    - account: Expenses:Transportation:Transit
      amount: "90.00"
    - account: Assets:Checking
#+end_src

Pros:
- No text parser needed; maps directly to ~Ledger::Transaction~ fields.
- ~YAML::Tiny~ is already a dependency.

Cons:
- Breaking format change; existing ~.tpl~ files must be migrated.
- More verbose; loses the ledger-native syntax users already know.

Only worthwhile if templates are ever generated programmatically.

**** Path C: Keep strings, fix only the insertion point (recommended short-term)

Keep raw transaction strings.  Add a ~fromTemplate~ method that generates
occurrence dates, computes the ID key (see below), checks ~$self->{id}~, and
appends the raw string at ~ofxpos~ via ~update_file~.  Solves file degradation
without a transaction parser.

Cons:
- Not zero parsing: computing the ID key requires extracting the account name
  and amount from the first posting line (one regex, but still parsing).
- Format coupling: ~ledgertemp2.pl~'s ~makeid~ produces ~EC-TEMP-2026-04-20+90.00~
  while ~Ledger::makeid~ produces ~EC-2026/04/20+$90.00~ — different date
  separators, different ~$~ handling, different infix.  Previously written
  template IDs are baked into the ledger; changing the format requires either
  a migration or a dual-format lookup in ~$self->{id}~.
- No ~Ledger::Transaction~ object: cannot edit in place later, harder to test.

*** Dedup mechanism

~$self->{id}~ is populated during construction from ~ledger csv~ output:
~CSV.pm~ line 92 sets ~$ledger->{id}->{$csv{key}} = $csv{payee}~ for every
posting that carries an ~ID:~ tag.  Previously written template transactions
are already present in this hash.

For Path A/B, ~addTransaction~ calls ~makeid~ to generate the lookup key and
checks ~$self->{id}~ before writing — the same path used by all other
importers.

For Path C, the caller must reproduce ~ledgertemp2.pl~'s key formula
(~{initials}-TEMP-{YYYY-MM-DD}{±amount}~) from the raw string in order to
perform the same lookup.

*** Relationship to ledgertemp3.pl

~ledgertemp3.pl~ takes explicit begin/end dates and has no ID deduplication by
design — it regenerates the full range on every run.  It writes to a separate
planning register (~check.ldg~), not the main cleared ledger.  If ~check.ldg~
is not part of the main ledger managed by Ledger.pm, there is no file
degradation problem and no pressure to integrate it.  Defer unless ~check.ldg~
is added to the main ~!include~ tree.

** DONE Track separate insertion files for cleared, pending, and uncleared transactions :feature:
:PROPERTIES:
:ID: FR-014
:END:
:LOGBOOK:
- State "DONE" from "TODO" [2026-04-20]
:END:

Currently all imported transactions are inserted at a single ~ofxpos~
— the byte position of the first non-cleared transaction in ~ofxfile~
(or EOF).  ~ofxfile~ itself is fixed to the file containing the last
cleared transaction.  This means every new transaction, regardless of
state, lands at the same point, which breaks when the ledger is split
across files by state.

*** Goal

Support a ledger organised as:

#+begin_example
!include cleared.ldg
!include pending.ldg
!include uncleared.ldg
#+end_example

Each new transaction should be appended to the file that matches its
state, so the three files stay coherent over time.

*** Design

Replace the single ~ofxfile~/~ofxpos~ pair with three independent
~(file, pos)~ pairs, one per state.

**** Insertion point logic (single-file and multi-file)

For each state, the insertion point is the ~bpos~ of the first
transaction whose state sorts /after/ the target state in the sequence
cleared → pending → uncleared, within the target file.  If no such
transaction exists, the insertion point is EOF of the target file.
This keeps each section contiguous regardless of whether the sections
live in one file or three.

Concretely, within a given file:

| Inserting   | Insertion point                                          |
|-------------+----------------------------------------------------------|
| cleared     | ~bpos~ of first pending or uncleared txn; else EOF       |
| pending     | ~bpos~ of first uncleared txn; else EOF                  |
| uncleared   | EOF                                                      |

(This generalises the current ~getofxpos~ logic, which already uses
~bpos~ of the first non-cleared transaction as the single insertion
point.)

**** File routing

| State     | Target file                                     | Fallback        |
|-----------+-------------------------------------------------+-----------------|
| cleared   | file of last cleared txn                        | —               |
| pending   | file of last pending txn                        | uncleared file  |
| uncleared | file of last uncleared txn                      | cleared file    |

For single-file setups all three resolve to the same file and the
insertion points within that file keep the sections ordered.  For
multi-file setups (~!include cleared.ldg~ etc.) each state maps to its
own file and the insertion point is EOF of that file (since no
later-state transactions exist there).

**** Detection and fallback

No special ~!include~ scanning is required — file routing falls out
naturally from which file each state's last transaction lives in.  If a
state has no existing transactions the fallback column above applies.
If none of the fallbacks resolve (empty ledger), behaviour is unchanged
from today.

This applies globally — ~fromStmt~, ~scheduleAppend~, and any other
path that calls ~update()~.

*** Implementation

~getofxpos~ replaced by ~getinsertionpoints~, which computes three ~(file,
pos)~ pairs (~cleared_file~/~cleared_pos~, ~pending_file~/~pending_pos~,
~uncleared_file~/~uncleared_pos~).  ~ofxfile~/~ofxpos~ remain as
backward-compatible aliases for the cleared pair.

~addStmtTran~ now calls ~insertionFileFor($state)~ to select the target
file before scheduling an append, rather than always using ~ofxfile~.

~update_file~ replaced the single ~$ofx~/~ofxpos~ sentinel with a
~%sentinels~ hash (~pos => [states]~).  Each sentinel writes only its
own state's ~@append~ transactions; balance entries go at the cleared
sentinel.  Multiple states can share the same sentinel position (e.g.
single-file setup) and are all handled in one pass.

~checkpending~'s pending→cleared branch changed from
~scheduleAppend($candidate->{file})~ to
~scheduleAppend($self->{edit} || $candidate->{file})~ so the moved
transaction targets the cleared file in multi-file setups.

A pre-existing bug in ~ledgerCSV~ was fixed as a prerequisite:
~%(xact.id)~ resets to 1 for each ~!include~ sub-file, so consecutive
rows from different files with the same ID were merged into one
transaction.  The boundary check now uses ~id != $last_id || file ne
$last_file~.

Covered by ~t/test_fr014.pl~ (34 assertions: single-file and
multi-file scenarios).
